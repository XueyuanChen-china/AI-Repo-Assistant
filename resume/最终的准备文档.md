### 1. 这个项目到底是什么？

#### 回答

这个项目是一个面向代码仓库场景的 Web AI 助手。  
用户可以先导入本地项目文件夹，浏览文件树，打开文件查看代码，然后选择几个文件作为上下文去提问，或者直接请求 AI 生成代码修改建议。  
如果模型返回的是结构化代码建议，系统不会只把它显示成一段文本，而是会继续生成 Diff，给用户逐文件审阅，最后由用户确认后再应用修改。

所以它和普通 AI 聊天页最大的区别是：

- 它理解的是代码仓库上下文
- 它支持代码修改建议
- 它有 Diff 审阅和审批应用闭环

### 2. 为什么做成 Web，而不是 CLI / VSCode 插件？

#### 回答

我第一版做成 Web，不是因为 Web 比 CLI 或插件更原生，而是因为它更适合首版快速验证核心交互。  
这个项目最重要的交互是文件树浏览、上下文选择、代码问答、Diff 审阅和审批应用，这些能力用 Web 表达更直观，也更适合演示和快速迭代。

所以我的判断是：

- 首版先用 Web 把核心交互闭环做出来
- 后续如果继续演进，再往 CLI 或插件这种更贴近真实开发工作流的形态升级

### 一句话版

Web 是首版验证核心交互的形态，不是说它比 CLI 更强。

### 3. 这个项目最核心的功能链路是什么？

#### 回答

这个项目最核心的链路，不是“用户问一句，模型回一段代码”，而是一个完整的仓库级代码建议闭环。

完整流程是：

1. 用户先选择本地项目文件夹，前端基于 File System Access API 构建文件树并支持代码预览。  
2. 用户勾选若干文件作为当前问题的上下文，然后在聊天区输入问题或修改请求。  
3. 前端会先读取这些选中文件的内容，连同当前问题和最近几轮有效历史消息一起组装成请求体，发送给后端。  
4. 后端收到请求后，不会直接把问题丢给模型，而是先经过 `contextBuilder + promptBuilder + chatOrchestrator` 这条链路：
   - `contextBuilder` 控制上下文预算
   - `promptBuilder` 组织系统提示、历史消息和文件内容
   - `chatOrchestrator` 调模型并解析 suggestion
5. 返回时优先走 SSE，前端先流式显示文本分析；如果后端成功提取出结构化代码建议，就会同时返回 `pendingSuggestions` 和 `diffPreviews`。  
6. 右侧查看器切到 Diff 模式，用户可以按文件查看每个修改建议的差异。  
7. 用户点击“应用修改”后，前端基于文件句柄把新内容真正写回本地文件，形成“建议生成 -> 审阅 -> 应用修改”的完整闭环。

### 4. 多文件上下文和多文件建议是怎么做的？

#### 回答

这个项目不只是支持单文件问答。  
用户可以勾选多个文件作为上下文，前端会把这些文件的路径和内容一起带给后端。后端组织 prompt 时，会把这些文件作为“本轮问题的仓库上下文”一起提供给模型。

如果模型返回多个文件建议，系统不会做成一键全量提交，而是把它们拆成：

- `pendingSuggestions[]`
- `diffPreviews[]`

前端再通过 `activeSuggestionIndex` 控制当前右侧展示哪一个文件的建议和 diff。

这样做的好处是：

- 支持跨文件理解和修改建议
- 又能保持逐文件审阅和逐文件应用的可控性

### 5. Diff 是怎么做的？为什么后来换成 Myers？

#### 回答

这个项目里的 Diff 不是简单把 before 和 after 并排显示，而是围绕“代码建议审阅”这条链专门做的。

流程是：

1. 后端先从模型返回中提取结构化 suggestion  
2. 再结合原文件内容生成 `diffPreview`
3. 前端收到后切到 Diff 模式
4. 用户按文件查看修改前后的差异，再决定是否应用

算法上，前端最后采用的是按行粒度的 Myers Diff。  
它的核心思路是先把修改前后的代码按行拆开，再求最短编辑路径，生成：

- `equal`
- `add`
- `remove`

最后再把这些操作转成可渲染的 diff rows。

之所以从更简单的方案换成 Myers，是因为早期那种轻量差异判断更适合 demo，但在连续插入、多行删除和相似代码块场景下容易不稳定。  
Myers 更接近 Git 的 diff 思路，在代码场景下展示结果更自然。

- 后端生成 diff 数据：[chatOrchestrator.ts](/d:/claude-code/claude-code-rev/ai-repo-assisant/server/src/services/chatOrchestrator.ts)
- 前端 diff 展示与算法：[InspectorPanel.tsx](/d:/claude-code/claude-code-rev/ai-repo-assisant/web/src/components/InspectorPanel.tsx)

Diff 让 AI 建议从“聊天文本”变成“可审阅修改”，而 Myers 让这个展示更接近真实代码工具。

### 6. 为什么不是一键全量审批，而是逐文件审批？

#### 回答

因为 AI 的多文件修改天然有误改风险。  
我第一版的设计重点不是追求最激进的自动化，而是先保证输出可控。

所以我采用的是：

- 多文件建议列表
- 逐文件 Diff 审阅
- 逐文件应用修改

这样用户可以一项一项确认每个文件的改动，避免一次性把错误修改全部写进仓库。  
后续如果继续演进，可以在这个基础上再补“全部应用”按钮，但第一版我有意把审批粒度控制在单文件级别。

### 一句话版

不是做不出一键全量，而是第一版优先控制风险和保证可控性。

### 7. 轻量多轮对话和提示词工程是怎么做的？

#### 回答

轻量多轮这部分，我做的不是复杂记忆系统，而是“最近几轮有效消息裁剪”。

每次用户发起新问题时，前端会从消息列表里筛出最近若干条有效历史消息，过滤掉欢迎语、系统提示和无意义内容，再把这些历史消息一起发给后端。  
后端在 `promptBuilder` 里把这些历史消息和当前问题、文件上下文一起组织成最终 prompt。

提示词工程这部分，我没有把它理解成“写几句 prompt 文案”，而是拆成三层：

- `contextBuilder`：控制哪些文件内容能进上下文、长度怎么裁剪
- `promptBuilder`：组织系统提示、历史消息和代码上下文
- `chatOrchestrator`：负责模型调用和 suggestion 结果提取

这样做的目的，是提升输出稳定性和 suggestion 的结构化概率，而不是只做一个大字符串拼接。

### 9. SSE 流式输出是怎么走的？

#### 回答

前端发起聊天时，优先请求 `/api/chat/stream`。  
后端会把返回过程拆成几类 SSE 事件：

- `context`
- `chunk`
- `done`
- `error`

前端收到流后，会用 `ReadableStream + TextDecoder` 逐段读取，按行解析 JSON，再用 Zod 校验事件结构。  
在流式过程中：

- `chunk` 事件会不断追加 assistant 消息内容
- `done` 事件会一次性落下 `reply`、`diffPreviews`、`pendingSuggestions` 和 `contextMeta`

这样前端既能先给用户流式反馈，又能在最终阶段把 suggestion 和 diff 状态完整挂上。

### 对应代码

- SSE 路由：[chat.ts](/d:/claude-code/claude-code-rev/ai-repo-assisant/server/src/routes/chat.ts)
- 前端流式读取：[chatApi.ts](/d:/claude-code/claude-code-rev/ai-repo-assisant/web/src/services/chatApi.ts)
- 消息状态更新：[useWorkspaceStore.ts](/d:/claude-code/claude-code-rev/ai-repo-assisant/web/src/store/useWorkspaceStore.ts)

### 一句话版

SSE 不是只把文本一段段显示出来，而是按事件协议驱动聊天文本、diff 和 suggestion 状态更新。

### 10. 为什么线上前后端要分开部署？

#### 回答

本地开发时，前端之所以能直接请求 `/api/*`，是因为 Vite 在开发环境里帮我做了 proxy。  
但线上部署后，这个代理环境不存在了，所以 Fastify 后端必须单独部署出去，前端再通过真实后端地址去请求。

我最后的做法是：

- 前端部署到 Vercel
- Fastify 后端部署到 Render
- 前端通过 `VITE_API_BASE_URL` 请求真实后端地址

这样本地和线上就能同时兼容：

- 本地走 Vite proxy
- 线上走真实 API 地址

### 一句话版

本地能用 `/api` 是因为有 Vite proxy，线上没有，所以前后端必须拆开部署。
