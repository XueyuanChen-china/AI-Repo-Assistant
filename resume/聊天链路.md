1. 前端从哪里开始发消息？
入口在：

FolderPickerWorkspacePage.tsx
核心函数是：

handleSendMessage()
它负责做几件事：

读取输入框内容
从 store 里拿当前选中的上下文文件
取最近几条有效历史消息
构建请求体
调用流式聊天接口
你可以把它理解成：

聊天发送按钮背后的总入口。

2. 前端怎么构建请求体？
在：

chatApi.ts
关键函数是：

buildChatRequestPayload(...)
它会把请求组织成类似这种结构：

{
  message,
  selectedPaths,
  contextFiles,
  historyMessages
}
这里最关键的是：

contextFiles
不是只传文件路径，而是会把选中的文件内容也一起带过去。
这样后端不需要再回头向前端要“这个文件到底写了什么”。

historyMessages
只保留最近几轮有效消息，不是把整个聊天历史全塞进去。
这就是你简历里写的“轻量多轮上下文”。

所以这一步解决的是：

如何把“聊天输入”升级成“仓库级问答输入”。

3. 后端怎么做上下文构建？
在：

contextBuilder.ts
这层的职责不是“生成 prompt”，而是先做：

上下文预算控制。

它主要控制三件事：

最多选多少个文件
单个文件最多截多少内容
总上下文最多放多少字符
你现在这个项目的思路是：

最多保留有限数量的上下文文件
每个文件有单独长度上限
总预算也有限制
如果超了，就会裁剪并记录哪些路径被截断
所以 contextBuilder 解决的问题是：

别让 prompt 因为文件太多、内容太长而无限膨胀。

4. 提示词怎么构建？
在：

promptBuilder.ts
这里会把三类信息组织成最终的模型输入：

系统提示
最近几轮历史消息
当前问题对应的文件上下文
你可以把它理解成：

把“原始材料”整理成模型真正能吃的 messages。

它做的事情包括：

告诉模型这是一个代码仓库助手
约束模型如何输出 suggestion
在需要修改代码时，尽量按约定格式返回结构化结果
把历史消息和文件上下文插入到正确位置
所以你面试里可以说：

我没有把 prompt 写成一大段拼接字符串，而是把上下文裁剪和提示词组织拆成了 contextBuilder + promptBuilder 两层，这样更容易控制预算，也更方便后续演进。

这个回答会很加分。

5. 后端怎么真正调模型？
在：

chatOrchestrator.ts
aiService.ts
这里是整条后端链路最核心的部分。

chatOrchestrator
它负责：

调用 contextBuilder
调用 promptBuilder
调用模型服务
解析模型返回
提取 suggestion
生成 diffPreviews 和 pendingSuggestions
它不是单纯“转发请求”，而是在做：

单轮编排。

这也是你项目里最像 Claude Code 思路的部分。

aiService
它负责：

真正向 DashScope 发请求
普通模式请求
SSE 流式请求
所以可以这样分工理解：

aiService：模型调用层
chatOrchestrator：业务编排层
6. SSE 是怎么走的？
后端路由在：

chat.ts
前端流式读取在：

chatApi.ts
后端
后端会把流式过程拆成几类事件：

context
chunk
done
error
它们最终会被序列化成一行一行 JSON，通过 SSE 发给前端。

比如逻辑上类似：

reply.raw.write(`data: ${JSON.stringify(event)}\n\n`)
前端
前端会：

用 fetch 拿到流式响应
用 ReadableStream + TextDecoder 去拆 chunk
按行解析
对每一行先 JSON.parse
再用 chatStreamEventSchema.parse(...) 校验事件结构
也就是说这段：

handlers.onEvent(chatStreamEventSchema.parse(JSON.parse(line)))
含义是：

JSON.parse(line)：把字符串变成对象
chatStreamEventSchema.parse(...)：用 Zod 校验并补默认值
再把合法事件交给前端状态层处理
所以 SSE 这条链不是“拿到文本就拼上去”，而是：

按事件协议驱动前端状态更新。

7. 前端怎么把 SSE 事件显示到聊天区？
主要在：

FolderPickerWorkspacePage.tsx
useWorkspaceStore.ts
流式过程中，前端会按事件类型做不同处理：

chunk
调用类似：

appendAssistantStreamChunk(...)
把流式文本一段段追加到当前 assistant 消息里。

done
调用类似：

finishAssistantMessage(...)
把最终的：

reply
diffPreviews
pendingSuggestions
contextMeta
一次性落到 store。

这一步很重要，因为：

聊天区文字是流式拼出来的
但右侧 diff 和 suggestion 状态，是在 done 时统一落下来的
8. 上下文元信息是怎么回来的？
还是在这条 SSE done 里。

后端会把：

实际用了哪些上下文文件
哪些路径因为预算被裁掉了
一起打包成：

contextMeta
前端收到后可以决定要不要显示“本轮实际用了哪些文件”。

所以这个项目里不仅有“上下文构建”，还有：

上下文可解释性。

一段适合面试直接说的口语版
这个项目的聊天链路是前端先把当前问题、选中的文件内容和最近几轮有效历史消息一起组装成请求体，再交给后端。后端不会直接把原始内容丢给模型，而是先经过 contextBuilder 做上下文预算控制，再由 promptBuilder 组织系统提示、历史消息和代码上下文，最后由 chatOrchestrator 调模型并解析 suggestion。返回时优先走 SSE，前端一边收 chunk 一边更新聊天内容，最终在 done 事件里把 reply、diff 和 suggestion 一起写入状态，所以它不是普通聊天页，而是带仓库上下文和结构化建议的聊天链路。
