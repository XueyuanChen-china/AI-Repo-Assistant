这个项目最核心的链路，不是“用户问一句，模型回一段代码”这么简单，而是一个完整的仓库级代码建议闭环。

整体流程是：

用户先选择本地项目文件夹，前端基于 File System Access API 构建文件树并支持代码预览。
用户勾选若干个文件作为当前问题的上下文，然后在中间聊天区输入问题或修改请求。
前端会先读取这些选中文件的内容，连同当前问题和最近几轮有效历史消息一起组装成请求体，发送给后端。
后端不会直接把问题丢给模型，而是先经过 contextBuilder + promptBuilder + chatOrchestrator 这条编排链：
contextBuilder 负责做上下文预算控制
promptBuilder 负责组织系统提示、历史消息和文件内容
chatOrchestrator 负责真正调用模型并解析 suggestion
前端通过 SSE 流式接收模型回复，先显示普通文本分析；如果后端成功提取出结构化代码建议，就会同时返回 diffPreviews 和 pendingSuggestions。
右侧查看器接到这些 suggestion 后，会切换到 Diff 模式，按文件展示修改前后差异。
用户可以逐文件审阅，点击“应用修改”后，前端基于文件句柄把新内容真正写回本地文件，形成“建议生成 -> 审阅 -> 应用修改”的完整闭环。
所以它和普通聊天页最大的区别是：
模型输出不是直接结束在聊天框里，而是会被组织成一个可审阅、可确认、可应用的代码修改流程。

对应代码链路
1. 用户发送消息
入口在：

FolderPickerWorkspacePage.tsx

handleSendMessage()
它负责：

读取输入框内容
取最近几条有效历史消息
调用 buildChatRequestPayload(...)
发起 streamChatRequest(...)
2. 前端组装请求体
在：

chatApi.ts
核心函数：

buildChatRequestPayload(...)
它会把这些内容拼成请求：

message
selectedPaths
contextFiles
historyMessages
也就是说，发给后端的不是一句裸问题，而是：

当前提问 + 选中文件内容 + 最近历史消息

3. 后端编排模型请求
在：

chatOrchestrator.ts
contextBuilder.ts
promptBuilder.ts
你可以这样理解：

contextBuilder：控制“哪些文件内容能进上下文、长度怎么裁剪”
promptBuilder：控制“怎么把系统提示、历史消息、代码上下文组织成最终 prompt”
chatOrchestrator：控制“怎么调模型、怎么提 suggestion、怎么生成 diff 结果”
4. SSE 流式返回
后端路由在：

chat.ts
前端流式读取在：

chatApi.ts
前端会接收几类事件：

context
chunk
done
error
其中最关键的是 done，它会带回来：

reply
diffPreviews
pendingSuggestions
contextMeta
5. 右侧显示 Diff
还是在：

FolderPickerWorkspacePage.tsx
useWorkspaceStore.ts
InspectorPanel.tsx
当 done 返回里有 diffPreviews 时：

store 会把 suggestion 和 diff 数据存起来
右侧 InspectorPanel 切到 Diff 模式
按 activeSuggestionIndex 展示当前文件的差异
6. 审批并写回本地文件
入口还是：

FolderPickerWorkspacePage.tsx

handleApplySuggestion(...)
真正写文件在：

函数：

writeSelectedRepoFile(...)
这里走的是前端文件句柄写回，不是后端直接改文件。

最后一句总结版
如果面试里想收尾，你可以这样说：

所以这个项目的核心不是“问一句话返回一段代码”，而是“前端读取仓库上下文 -> 后端组织 prompt 和 suggestion -> SSE 返回分析结果 -> 前端展示 Diff -> 用户审批后写回本地文件”的完整闭环，这也是它和普通 AI 聊天页最大的区别。
